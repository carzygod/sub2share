## 3. 已发现问题

### 3.1 购买租赁开通失败

复测路径：

1. 注册测试账号。
2. mock 充值 25 USD。
3. 购买 Codex 标准月租套餐。

修复前结果：

- 账号注册成功。
- 钱包充值成功。
- 创建订单失败，API 返回 `internal_error`。
- `zyz-api` 日志显示 Sub2API 返回：

```text
Sub2API request failed: 401 {"code":"INVALID_TOKEN","message":"Invalid token"}
```

根因：

- 业务平台 `.env` 中 `SUB2_ADMIN_TOKEN` 为空。
- 业务平台 `.env` 中配置了 `SUB2_ADMIN_EMAIL` / `SUB2_ADMIN_PASSWORD`，但该账号无法登录 Sub2API。
- Sub2API 数据库中实际管理员账号为 `carzyg@outlook.com`。
- 业务平台无法获取 Sub2 管理员 access token，因此无法调用 `/api/v1/admin/users` 创建托管用户。

### 3.2 服务器部署版本落后于本地仓库

远端 `/opt/zhisuan-yizhan/user` 不是 git 工作区，像是直接拷贝部署。远端源码与本地仓库存在明显差异：

| 项目 | 远端服务器 | 本地仓库 |
| --- | --- | --- |
| 用户注册 | 邮箱 + 密码注册可用 | 密码注册已禁用 |
| 用户登录 | 邮箱 + 密码登录 | 用户侧 Google / X OAuth |
| Prisma 迁移 | 仅 `0001_init` | 有 `0001_init` 和 `0002_oauth_identities` |
| OAuth 配置 | `.env` 未配置 Google/X | 源码需要 Google/X OAuth 配置 |
| 部署方式 | systemd + vite preview | 仓库有 Dockerfile / Compose，也可 systemd 部署 |

该差异意味着不能直接把本地当前版本覆盖到服务器，否则会涉及数据库迁移、登录方式变化、环境变量补充和前端入口变化。

### 3.3 Sub2API 上游资源池存在不可用记录

Sub2API 日志中出现多条：

```text
openai.account_select_failed ... error="no available accounts"
```

这说明 Sub2API 本身虽然健康，但某些实际模型请求没有可用上游账号。该问题与业务平台创建 Key 的故障不同，需要在 Sub2API 资源池中补充或恢复可用上游账号。

### 3.4 本地源码存在账务幂等风险

本地仓库 `sync-sub2-usage.ts` 原逻辑对 `UsageRecord` 使用 `upsert`，但后续扣钱包和创建结算未判断该 usage 是否已存在。重复同步同一 Sub2 usage 时可能重复扣费、重复创建结算。

## 5. 本地源码修复

### 5.1 用量同步幂等

修改文件：

```text
user/apps/api/src/jobs/sync-sub2-usage.ts
```

修复内容：

- 同步结果返回 `imported`、`skipped`、`unmatched` 统计。
- 事务内先检查 `sub2RequestId` 是否已存在。
- 已存在 usage 直接跳过，不再扣钱包、不再创建结算。
- 捕获唯一键冲突 `P2002`，并视为已同步跳过，抵御并发重复同步。
- 钱包余额不足时不扣成负数，将 usage 标记为 `pending`，并将租赁状态改为 `low_balance`。
- usage 钱包流水的 `refId` 改为本地 `UsageRecord.id`，方便对账。

### 5.2 订单扣款流水补充订单 ID

修改文件：

```text
user/apps/api/src/modules/orders/routes.ts
```

修复内容：

- 原订单扣款流水只有 `refType: "order"`，没有 `refId`。
- 现在先创建订单，再创建钱包流水，并写入 `refId: order.id`，方便退款、对账和追踪。

## 6. 仍待处理事项

### 6.1 服务器版本升级策略

服务器当前是旧版密码注册系统，本地仓库是新版 OAuth 系统。升级前必须决定：

1. 是否保留远端邮箱密码注册/登录。
2. 是否迁移到 Google/X OAuth。
3. 是否要为远端数据库执行 `0002_oauth_identities`。
4. 是否需要迁移已有用户的登录方式。
5. 是否改用 git 工作区部署，避免继续手工拷贝。

建议先将服务器 `/opt/zhisuan-yizhan/user` 纳入可追踪发布流程，再做大版本切换。

### 6.2 Sub2API 资源池可用性

Sub2API 存在 `no available accounts` 日志。需要检查：

1. 上游账号是否过期、限流、禁用或未分配到正确 group。
2. 用户 API Key 所在 group 是否有可调度账号。
3. 请求模型是否被资源池支持。
4. 上游账号健康检查是否正常。

### 6.3 线上数据库清理

巡检过程中创建了 `codex_health_` / `codex_health_after_` 前缀测试用户，并产生测试充值、订单和租赁。建议在确认不再需要复盘后清理这些测试数据，或保留作为健康检查样本。

### 6.4 生产级 P0 改造

需要继续实现：

1. 真实支付替换 mock 充值。
2. 订单创建幂等键。
3. 定时用量同步和同步 cursor。
4. 租赁到期自动禁用。
5. 欠费自动限制 / 暂停。
6. 后台资源审核、上线、禁用。
7. 操作审计和生产 CORS 白名单。
8. OAuth state 迁移到 Redis。

## 7. 当前结论

线上服务已经恢复核心购买链路：用户可以注册、充值、购买套餐，业务 API 可以成功调用 Sub2API 创建托管用户和 API Key。

当前最大剩余风险不是服务进程不可用，而是版本治理和生产化能力不足：服务器部署版本落后于本地仓库，且 Sub2API 资源池仍可能没有可用上游账号。下一步应先确定升级策略，再将本地已修复的账务幂等逻辑安全发布到服务器。
