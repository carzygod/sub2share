# OpenAI/Codex 反代速率窗口清理

实现日期：2026-06-10

## 背景

本地 `/v1/*` OpenAI/Codex 反代使用租赁级 60 秒滚动窗口执行 RPM/TPM 闸门。窗口状态保存在 API 进程内，适合当前单实例部署，但如果只追加窗口而不清理长期不用的租赁 ID，长时间运行后内存中的速率窗口表会逐步增长。

这类增长不会直接影响单次请求正确性，但会降低长期运行稳定性，也会让后续迁移到共享限流器时更难判断真实活跃窗口规模。

## 已实现范围

- 新增 `pruneProxyRateLimitWindow()` helper，统一按滚动窗口清理过期请求和 token 事件。
- 新增 `isProxyRateLimitWindowEmpty()` helper，用于判断窗口是否已经没有任何有效事件。
- `evaluateProxyRateLimitWindow()` 复用统一清理逻辑，保证每次 RPM/TPM 判断只基于当前有效窗口。
- 反代路由新增节流的全局窗口表清理：
  - 最多每 60 秒扫描一次 `proxyRateWindows`。
  - 每个窗口先裁掉过期事件。
  - 裁剪后为空的租赁窗口会从 Map 中删除。
- 新增自动化测试覆盖过期请求和 token 事件的裁剪，以及空窗口识别。

## 管理员价值

- 长时间运行的 API 进程不会因为历史租赁请求不断堆积无效速率窗口。
- 速率限额、并发限额和反代日志继续保持单实例下的实时保护能力。
- 后续如果扩展到多实例，可以更明确地把“当前 60 秒内仍活跃的窗口”迁移到 Redis 或网关级限流器。

## 边界

该清理只处理本地进程内 RPM/TPM 状态，不会修改数据库中的租赁、订单、余额或请求日志。多实例部署时，各实例仍会各自维护窗口；需要共享限流器才能实现跨实例一致限流。
