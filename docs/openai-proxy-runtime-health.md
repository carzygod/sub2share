# OpenAI/Codex 反代运行态巡检

实现日期：2026-06-10

## 背景

本地 `/v1/*` OpenAI/Codex 反代已经具备 API Key、租赁、余额、请求量、RPM/TPM、并发和上游转发等多层门禁。管理员此前可以通过反代请求日志查看历史请求结果，但无法直接看到当前 API 进程内是否存在活跃并发租约、速率窗口，以及当前限流器是否仍是进程内作用域。

这会影响生产可用性判断：单实例部署下进程内限流器可以提供即时保护；如果 API 横向扩容，多实例会各自计数，管理员需要在巡检中明确看到这一边界。

## 已实现范围

- 新增 `inspectOpenAiProxyRuntime()` helper，用于把 OpenAI 反代运行态指标转成巡检摘要和问题样本。
- `/v1/*` 反代路由新增 `inspectOpenAiProxyRuntimeState()`：
  - 强制裁剪过期 RPM/TPM 事件后再生成快照。
  - 返回当前进程活跃并发租赁数和并发租约数。
  - 返回当前进程活跃速率窗口租赁数、请求事件数、token 事件数和估算 token 总量。
  - 返回 60 秒速率窗口、清理间隔和最近清理时间。
- `GET /api/admin/system-health` 新增检查项：
  - `openAiProxyRuntime`
  - 展示名：`OpenAI 反代运行态`
  - 生产环境下进程内限流器标记 warning，提示单实例部署或后续迁移到 Redis/网关级共享限流器。
- 自动化测试覆盖生产环境进程内限流器的 warning 判定。

## 管理员价值

- 管理员可以在 `可用性巡检` 中直接看到当前 API 进程是否正在承载 OpenAI/Codex 反代并发和速率窗口。
- 售后排障时，可以把最近 1 小时反代请求日志和当前运行态指标放在一起判断，区分历史异常和实时承压。
- 生产部署时，可以更早发现“多实例部署但限流器仍是进程内”的架构风险。

## 边界

本次巡检只暴露当前 API 进程内状态，不会把并发或 RPM/TPM 状态迁移到 Redis，也不会改变请求拦截行为。完整多实例一致限流仍需要后续把并发租约和速率窗口迁移到共享存储或网关级限流器。
